1. Contexto Executivo

1.1 Apresentação

O Sistema Talentos Públicos Recife - Trajetórias é uma solução digital desenvolvida para modernizar e dar transparência ao processo seletivo de servidores públicos municipais que pleiteiam afastamento remunerado para cursar mestrado stricto sensu. O sistema substitui processos físicos por um fluxo 100% digital, garantindo auditabilidade, padronização e agilidade na análise das candidaturas.

O portal permite a inscrição, envio de documentação comprobatória, acompanhamento de status e interposição de recursos, tudo integrado às bases de dados corporativas da Prefeitura.

1.2 Público Alvo

  • Servidores Efetivos: Usuários principais que submetem candidaturas.
  • Gestores de RH/SAD: Responsáveis pela validação cadastral.
  • Comissão Avaliadora: Banca examinadora que pontua os projetos (Plano de Aplicação).
  • Auditores: Órgãos de controle que verificam a lisura do processo.

1.3 Guia de Usabilidade

A jornada do usuário foi desenhada para ser linear e autoexplicativa:

  1. Acesso e Identificação: O servidor acessa o portal logado (SSO/Token), onde seus dados funcionais (Cargo, Matrícula, Lotação) são carregados automaticamente via integração, evitando erros de preenchimento.
  2. Formulário de Inscrição:
    • Dados Complementares: Preenchimento de raça/cor, gênero, contatos e informações não constantes na base RH.
    • Requisitos Básicos: Checklist obrigatório de elegibilidade (estabilidade, tempo de serviço, etc).
  3. Documentação: Upload de arquivos PDF (Diploma, Carta de Aceite, Carta de Motivação, Plano de Aplicação). O sistema valida extensões e obrigatoriedade.
  4. Dados Acadêmicos: Cadastro dinâmico de até 3 universidades pretendidas com seus respectivos cursos.
  5. Declaração e Envio: Aceite legal dos termos do edital e submissão final, gerando número de protocolo.

2. Especificação Técnica

2.1 Stack Tecnológico

O projeto segue os padrões de desenvolvimento web moderno com foco em performance e manutenibilidade:

  • Frontend: HTML5 Semântico, Dashboard Layout.
  • Estilização: CSS3 Core Variables + Bootstrap 5.3 (Grid System e Componentes UI). Fonte Public Sans.
  • Scripting: JavaScript ES6+ (Vanilla) para manipulação de DOM, validação de formulários (Constraints API) e construção de objetos JSON.
  • Backend/Middleware: n8n (Workflow Automation) para orquestração de chamadas.
  • Integração: RESTful API via Fetch API (Client-side) e Webhooks.

2.2 Decisões de Design

A interface utiliza um layout de sidebar fixa com navegação por abas (SPA-like behavior), permitindo transição fluida entre as seções "Inscrição", "Recursos" e "Acompanhamento" sem recarregamento de página. Utiliza-se localStorage e variáveis de sessão para persistência temporária de dados durante o preenchimento.

5. Fluxo de Processamento

O ciclo de vida da informação transita entre a camada de apresentação (Browser) e a camada de persistência/notificação, orquestrado para garantir a atomicidade da transação.

Etapa 1: Inicialização e Carregamento (Load)

Ao carregar a página arealogada.html, o sistema intercepta as variáveis de ambiente (injetadas pelo servidor de aplicação ou gerenciador de contexto) contendo os dados do servidor autenticado. Scripts de correção (fix-payload-structure.js) normalizam a estrutura JSON de entrada para garantir compatibilidade.

Etapa 2: Interatividade e Validação (Client-Side)

O usuário interage com o formulário. O JavaScript monitora eventos change e input. Validações síncronas ocorrem em tempo real:

  • Verificação de obrigatoriedade de campos (HTML5 Constraint Validation).
  • Regras de negócio condicionais (ex: Aparecer campos de "Outro Cargo" somente se selecionado "Sim").
  • Controle de arquivos: Validação de extensão (.pdf) e tamanho máximo.

    • Etapa 3: Construção do Payload e Envio

    Ao acionar "Enviar Inscrição":

    1. O sistema coleta todos os valores dos inputs.
    2. Codifica os arquivos PDF em Base64 (assíncrono).
    3. Monta o objeto JSON consolidado (conforme item 4.1).
    4. Dispara requisição POST para o endpoint de Webhook configurado.

    Etapa 4: Processamento Assíncrono (Server-Side)

    O Webhook recebe o JSON, valida a integridade do token de sessão e inicia a transação no banco de dados. Após persistir as tabelas TB_TP_*, dispara e-mail de confirmação para o servidor com o número do protocolo gerado.
